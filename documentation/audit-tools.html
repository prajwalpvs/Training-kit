Introduction

These guidelines are based on the Web Content Accessibility Guidelines (WCAG) 2.1, which is the internationally recognized standard for web accessibility. WCAG is organized around four principles:

  • Perceivable: Information and user interface components must be presentable to users in ways they can perceive.
  • Operable: User interface components and navigation must be operable.
  • Understandable: Information and the operation of the user interface must be understandable.
  • Robust: Content must be robust enough that it can be interpreted by a wide variety of user agents, including assistive technologies.

These guidelines cover WCAG 2.1 success criteria across three levels of conformance:

  • Level A: The most basic web accessibility features. Failure to meet these criteria means some users will find it impossible to access your content.
  • Level AA: Addresses the most common barriers for users with disabilities. Most organizations aim for this level compliance. Many laws and regulations reference this level.
  • Level AAA: The highest level of accessibility that provides enhanced accessibility for users with specific needs.

How to Use These Guidelines

These guidelines are organized by the WCAG principles and success criteria. Each section includes:

  • A description of the guideline
  • Practical implementation advice
  • Examples of good and bad practices
  • Code samples
  • Testing tips

If you're new to accessibility, we recommend starting with Level A criteria and progressively enhancing your content to meet higher levels.

1. Perceivable

Information and user interface components must be presentable to users in ways they can perceive. This means that users must be able to perceive all relevant information with at least one of their senses.

1.1 Text Alternatives

Non-text Content

WCAG 1.1.1 Level A

All non-text content that is presented to the user has a text alternative that serves the equivalent purpose, except for the situations listed below.

  • Controls, Input: If non-text content is a control or accepts user input, then it has a name that describes its purpose.
  • Time-Based Media: If non-text content is time-based media, then text alternatives at least provide descriptive identification of the non-text content.
  • Test: If non-text content is a test or exercise that would be invalid if presented in text, then text alternatives at least provide descriptive identification of the non-text content.
  • Sensory: If non-text content is primarily intended to create a specific sensory experience, then text alternatives at least provide descriptive identification of the non-text content.
  • CAPTCHA: If the purpose of non-text content is to confirm that content is being accessed by a person rather than a computer, then text alternatives that identify and describe the purpose of the non-text content are provided, and alternative forms of CAPTCHA using output modes for different types of sensory perception are provided to accommodate different disabilities.
  • Decoration, Formatting, Invisible: If non-text content is pure decoration, is used only for visual formatting, or is not presented to users, then it is implemented in a way that it can be ignored by assistive technology.

Implementation Guidelines

  • Provide descriptive alt text for images that convey information
  • Use empty alt text (alt="") for decorative images
  • For complex images like charts or diagrams, provide extended descriptions in the surrounding text or via a link
  • Ensure audio and video content has text alternatives (transcripts, captions)
  • Use proper labels for form controls and interactive elements

Examples

Good Example

Image with descriptive alt text:

<img src="chart.png" alt="Bar chart showing sales growth of 25% in Q1, 15% in Q2, and 30% in Q3">

Decorative image with empty alt text:

<img src="decorative-divider.png" alt="" role="presentation">

Bad Example

Missing alt attribute:

<img src="chart.png">

Uninformative alt text:

<img src="chart.png" alt="Chart">

Redundant alt text:

<img src="chart.png" alt="Image of a chart">

Testing Tips

  • Use a screen reader to check if non-text content is properly conveyed
  • Disable images in your browser to ensure text alternatives provide equivalent information
  • Inspect the HTML to ensure all non-text elements have appropriate text alternatives
  • Use automated testing tools to identify missing alt text

1.2 Time-based Media

Captions (Prerecorded)

WCAG 1.2.2 Level A

Captions are provided for all prerecorded audio content in synchronized media, except when the media is a media alternative for text and is clearly labeled as such.

Implementation Guidelines

  • Provide synchronized captions for all video with audio
  • Ensure captions include all spoken dialogue
  • Include speaker identification when multiple people are speaking
  • Include relevant non-speech information (sound effects, meaningful music)
  • Use a video player that supports captions

Examples

Good Example

Video with captions:

<video controls> <source src="video.mp4" type="video/mp4"> <track kind="captions" src="captions.vtt" srclang="en" label="English" default> Your browser does not support the video tag. </video>

Bad Example

Video without captions:

<video controls> <source src="video.mp4" type="video/mp4"> Your browser does not support the video tag. </video>

Audio Description or Media Alternative (Prerecorded)

WCAG 1.2.3 Level A

An alternative for time-based media or audio description of the prerecorded video content is provided for synchronized media, except when the media is a media alternative for text and is clearly labeled as such.

Implementation Guidelines

  • Provide audio descriptions for videos that convey visual information essential to understanding the content
  • Audio descriptions should narrate important visual elements that are not conveyed through dialogue
  • Alternatively, provide a full text transcript that includes descriptions of all visual and auditory information

1.3 Adaptable

Info and Relationships

WCAG 1.3.1 Level A

Information, structure, and relationships conveyed through presentation can be programmatically determined or are available in text.

Implementation Guidelines

  • Use semantic HTML elements that reflect the intended structure of the content
  • Use proper heading structure (h1-h6) in a hierarchical order
  • Use lists (ul, ol, dl) for list-like content
  • Use tables for tabular data, with proper headers
  • Use fieldset and legend for grouping form elements
  • Associate form labels with their controls

Examples

Good Example

Proper heading structure:

<h1>Main Page Title</h1> <section> <h2>Section Heading</h2> <p>Section content...</p> <h3>Subsection Heading</h3> <p>Subsection content...</p> </section>

Proper form labeling:

<label for="name">Name:</label> <input type="text" id="name" name="name">

Bad Example

Fake headings:

<div style="font-weight: bold; font-size: 24px;">Main Page Title</div> <div style="font-weight: bold; font-size: 18px;">Section Heading</div>

Unlabeled form inputs:

<div>Name:<div>Name:</div> <input type="text" name="name">

1.4 Distinguishable

Use of Color

WCAG 1.4.1 Level A

Color is not used as the only visual means of conveying information, indicating an action, prompting a response, or distinguishing a visual element.

Implementation Guidelines

  • Always use additional indicators besides color (text, icons, patterns)
  • Ensure information is understandable when viewed in grayscale
  • For links within text, use underlines or other visual cues besides color
  • For form validation, use icons, text, and patterns in addition to color

Examples

Good Example

Error indication with color and icon/text:

<p class="error"> <span aria-hidden="true">‚ùå</span> <span class="error-text">Error: Please enter a valid email address</span> </p>

Bad Example

Using only color to indicate an error:

<p style="color: red;">Please enter a valid email address</p>

Contrast (Minimum)

WCAG 1.4.3 Level AA

The visual presentation of text and images of text has a contrast ratio of at least 4.5:1, except for the following:

  • Large Text: Large-scale text and images of large-scale text have a contrast ratio of at least 3:1;
  • Incidental: Text or images of text that are part of an inactive user interface component, that are pure decoration, that are not visible to anyone, or that are part of a picture that contains significant other visual content, have no contrast requirement.
  • Logotypes: Text that is part of a logo or brand name has no contrast requirement.

Implementation Guidelines

  • Ensure text has sufficient contrast against its background
  • Use contrast ratio calculators to verify compliance
  • For normal text, maintain at least 4.5:1 contrast ratio
  • For large text (18pt or 14pt bold), maintain at least 3:1 contrast ratio
  • Remember that contrast requirements apply to text on images, buttons, and form controls

2. Operable

User interface components and navigation must be operable. This means that users must be able to operate the interface (the interface cannot require interaction that a user cannot perform).

2.1 Keyboard Accessible

Keyboard

WCAG 2.1.1 Level A

All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints.

Implementation Guidelines

  • Ensure all interactive elements can be accessed and operated using the keyboard alone
  • Use semantic HTML elements for interactive components (<button>, <a>, etc.)
  • If using custom interactive elements, add keyboard event handlers
  • Ensure dropdown menus, modals, and other widgets are keyboard accessible
  • Test with keyboard only (no mouse) to verify accessibility

Examples

Good Example

Using semantic HTML:

<button type="button">Submit Form</button>

Custom element with keyboard support:

<div role="button" tabindex="0" onclick="submitForm()" onkeydown="if(event.key==='Enter'||event.key===' ')submitForm()" aria-label="Submit Form"> Submit </div>

Bad Example

Non-semantic element without keyboard support:

<div onclick="submitForm()">Submit</div>

Mouse-dependent interaction:

<div onmouseover="showMenu()" onmouseout="hideMenu()">Menu</div>

No Keyboard Trap

WCAG 2.1.2 Level A

If keyboard focus can be moved to a component of the page using a keyboard interface, then focus can be moved away from that component using only a keyboard interface, and, if it requires more than unmodified arrow or tab keys or other standard exit methods, the user is advised of the method for moving focus away.

Implementation Guidelines

  • Ensure users can navigate to and away from all interactive elements using the keyboard
  • Avoid creating "keyboard traps" where focus cannot be moved away from an element
  • For modal dialogs, trap focus within the dialog while it's open, but allow closing with ESC key
  • If non-standard keyboard navigation is required, provide instructions

2.4 Navigable

Bypass Blocks

WCAG 2.4.1 Level A

A mechanism is available to bypass blocks of content that are repeated on multiple web pages.

Implementation Guidelines

  • Provide a "Skip to main content" link at the beginning of the page
  • Use proper heading structure to allow navigation by headings
  • Use ARIA landmarks to define page regions
  • Provide a table of contents for long documents

Examples

Good Example

Skip link at the beginning of the page:

<a href="#main-content" class="skip-link">Skip to main content</a> <header> <!-- Navigation and header content --> </header> <main id="main-content"> <!-- Main content --> </main>

Using ARIA landmarks:

<header role="banner"> <!-- Site header content --> </header> <nav role="navigation"> <!-- Navigation --> </nav> <main role="main"> <!-- Main content --> </main> <footer role="contentinfo"> <!-- Footer content --> </footer>

Bad Example

No way to bypass repetitive navigation:

<div> <!-- Logo and site header --> </div> <div> <!-- Large navigation menu --> </div> <div> <!-- Main content --> </div>

3. Understandable

Information and the operation of the user interface must be understandable. This means that users must be able to understand the information as well as the operation of the user interface.

3.1 Readable

Language of Page

WCAG 3.1.1 Level A

The default human language of each web page can be programmatically determined.

Implementation Guidelines

  • Specify the primary language of the page using the lang attribute on the <html> element
  • Use valid language codes according to BCP 47 (e.g., "en" for English, "es" for Spanish)
  • This helps screen readers pronounce content correctly

Examples

Good Example

Specifying page language:

<html lang="en"> <!-- Page content --> </html>

Bad Example

No language specified:

<html> <!-- Page content --> </html>

Language of Parts

WCAG 3.1.2 Level AA

The human language of each passage or phrase in the content can be programmatically determined except for proper names, technical terms, words of indeterminate language, and words or phrases that have become part of the vernacular of the immediately surrounding text.

Implementation Guidelines

  • Use the lang attribute on elements containing content in a different language than the page's default language
  • This ensures screen readers pronounce foreign language content correctly

Examples

Good Example

Marking up foreign language phrases:

<p>The French expression <span lang="fr">c'est la vie</span> means "that's life."</p>

Bad Example

Foreign language without language indication:

<p>The French expression c'est la vie means "that's life."</p>

4. Robust

Content must be robust enough that it can be interpreted by a wide variety of user agents, including assistive technologies. This means that users must be able to access the content as technologies advance.

4.1 Compatible

Parsing

WCAG 4.1.1 Level A

In content implemented using markup languages, elements have complete start and end tags, elements are nested according to their specifications, elements do not contain duplicate attributes, and any IDs are unique, except where the specifications allow these features.

Implementation Guidelines

  • Write valid, well-formed HTML
  • Use proper opening and closing tags
  • Ensure proper nesting of elements
  • Avoid duplicate ID attributes
  • Validate your HTML using tools like the W3C Validator

Name, Role, Value

WCAG 4.1.2 Level A

For all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies.

Implementation Guidelines

  • Use semantic HTML elements whenever possible
  • Provide labels for form controls
  • Use appropriate ARIA roles, states, and properties when creating custom UI components
  • Ensure custom controls communicate state changes to assistive technologies

Examples

Good Example

Custom checkbox with proper ARIA:

<div role="checkbox" aria-checked="false" tabindex="0" aria-labelledby="custom-checkbox-label" onclick="toggleCheckbox(this)" onkeydown="if(event.key===' ')toggleCheckbox(this)"> <!-- visual checkbox indicator --> </div> <span id="custom-checkbox-label">Accept terms and conditions</span>

Bad Example

Custom checkbox without ARIA:

<div onclick="toggleCheckbox(this)" class="custom-checkbox"></div> <span>Accept terms and conditions</span>
